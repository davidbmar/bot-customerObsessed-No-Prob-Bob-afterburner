# Vision: No Prob Bob — Customer-Obsessed Agent Platform

## Product Name

**No Prob Bob** — Customer-Obsessed Agent Platform

## Problem Statement

AI developer tools assume you already know what to build. They start with the code — "generate a component," "write a function," "deploy this app." But the hardest part of building products isn't the engineering. It's understanding what your customers actually need.

The gap between "customer has a problem" and "engineer writes code" is filled by meetings, Slack threads, and tribal knowledge. Discovery interviews are manual, requirements flow through Notion/Jira, and context is lost at every handoff. No existing tool bridges customer conversations to automated engineering pipelines.

## Target Audience

**Product builders and solo founders** who talk to customers but struggle to translate conversations into engineering requirements. They know their customers have real problems but lack tooling to systematically capture, structure, and act on what they learn. They use Afterburner for engineering but need a front door that starts with the customer, not the code.

## Key Differentiators

- **Customer-first, not code-first.** The bot obsessively understands the problem before a single line is written — it won't accept "add a dark mode" without asking why.
- **Personality as a platform.** A reusable framework where bot behavior is encoded in markdown documents with inheritance (base → customer-discovery). Future personalities (tech-support, internal-PM, sales) plug into the same system. Principles, not scripts — the LLM uses judgment.
- **Multi-provider LLM.** Switch between Ollama (local/free/private), Claude (Haiku/Sonnet/Opus), or OpenAI models at runtime from the web UI. No restart required.
- **Agent SDK evolution path.** The TypeScript Agent SDK reference architecture (query loop, WebSocket streaming, permission gating) charts the path from a Python bot to a production agent platform.
- **Deep Afterburner integration.** Bot outputs (seed docs) become engineering inputs automatically. The full loop: Customer conversation → Seed docs → Vision → Plan → Sprint → Shipped code → Customer feedback.

## What Exists Today (Sprints 1–11)

After 11 sprints, No Prob Bob is a working product with **183 tests** and **22 shipped features**:

### Personality Framework
- Markdown-based personality documents with inheritance (base.md → customer-discovery.md)
- 9 discovery principles (Customer Obsession, Dive Deep, Insist on High Standards, Bias for Action, Data Driven, etc.)
- Personality hot-reload — change behavior without restarting the server
- Scenario-based evaluation framework (YAML test scenarios, CLI runner)

### Three Interfaces
- **Web Chat** (localhost:1203) — dark-themed UI with debug panel (principles, tools, tokens, latency, facts), settings panel (switch personality/model/provider), welcome message, progress indicator, conversation export, new chat button
- **Telegram** — polling transport, same brain as web chat
- **CLI** — `cli.py start` (server + polling), `cli.py chat` (interactive/single-message), `cli.py status`, `cli.py evaluate`

### Six Afterburner Tools
1. `save_discovery` — writes seed docs to a project
2. `get_project_summary` — what's been built, what's planned
3. `add_to_backlog` — adds features/bugs to project backlog
4. `get_sprint_status` — checks current sprint progress
5. `generate_vision` — synthesizes discovery into Vision doc
6. `feedback_on_sprint` — "here's what shipped, does it match?"

### Infrastructure
- Multi-provider LLM (Ollama + Claude via anthropic SDK) with runtime switching
- JSONL conversation memory with per-chat isolation
- Fact extraction (LLM-generated structured summaries)
- Multi-project support (switch between Afterburner projects)
- Conversation export to markdown
- Config system (~/.config/afterburner-bots/config.json)

## Where It's Going

### Agent SDK Migration (Phase 3)
The seed docs include a TypeScript Agent SDK reference implementation (vibecode-agent) that demonstrates the next architectural evolution:

- **Query loop pattern** — the Claude Agent SDK drives a multi-turn tool-calling loop with `query()`, replacing the current single-shot Ollama/Claude calls
- **WebSocket streaming** — real-time message streaming from agent to frontend, replacing HTTP request/response
- **Permission gating** — typed permission model (auto-allow safe ops, require human approval for destructive actions), replacing the current implicit trust model
- **In-process MCP server** — tools registered via `createSdkMcpServer()` with Zod schemas, enabling typed tool definitions

### Customer Profiles & Memory Evolution
- Cross-conversation customer knowledge (who said what across sessions)
- Structured profile building from extracted facts
- Context injection from previous conversations with the same customer

### Voice & Multi-Platform
- Whisper STT for Telegram voice messages
- TTS responses (optional)
- WhatsApp support via tool-telegram-whatsapp transport layer

### Second Personality Type
- Tech-support personality to validate the framework's reusability
- Personality marketplace concept — community-contributed bot behaviors

## Success Criteria

- Bot asks about users and use cases when given a feature request (not "what kind of feature?")
- 5-7 exchange conversations produce structured synthesis (Problem/Users/Use Cases/Success Criteria)
- save_discovery tool writes seed docs that Afterburner's lifecycle pipeline can consume
- Web chat debug panel shows principles, tools, memory, and latency
- Conversations don't bleed between sessions (memory isolation)
- Multi-provider LLM switching works at runtime without restart
- Agent SDK migration preserves all existing functionality while adding streaming and permissions

## FAQ

**Q: Why multiple LLM providers?**
A: Different situations call for different models. Ollama (Qwen 3.5) provides free, private, offline inference for development and sensitive customer data. Claude provides higher-quality reasoning for complex discovery sessions. The gateway pattern means the provider is a config choice, not an architecture choice.

**Q: What is the Agent SDK?**
A: Anthropic's Claude Agent SDK provides a `query()` loop that drives multi-turn tool calling, WebSocket streaming, and typed permission gating. The seed docs include a reference implementation (vibecode-agent) that shows how No Prob Bob would evolve from a Python bot into a TypeScript agent platform. This is the Phase 3 architectural target.

**Q: How does personality inheritance work?**
A: Personalities are markdown documents. `base.md` defines core values (be honest, be helpful, stay focused). `customer-discovery.md` extends base with domain-specific principles (Customer Obsession, Dive Deep). The `PersonalityLoader` reads both files and concatenates them into the LLM system prompt. New personalities inherit base behavior automatically — you only define what's different.

**Q: Can I use this without Afterburner?**
A: The bot's core (personality framework, LLM gateway, memory, web chat) works standalone. The Afterburner tools (save_discovery, generate_vision, etc.) are optional — they activate when pointed at an Afterburner project. Without them, you have a personality-driven chat bot. With them, you have an end-to-end discovery-to-engineering pipeline.
